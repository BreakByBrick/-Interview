Утилита предназначена для компрессии (кодирования) и декомпрессии (декодирования) файлов.
Кодирование/декодирование файла происходит поблочно, блоки байтов файла называются слайсами.

Обработка файла реализована в соответствии с паттерном Pipeline или Pipes and Filters.

Для реализации поставленной задачи необходимо 2 очереди обработки слайсов (pipes):

	1. Очередь для считанных слайсов, ожидающих кодирования/декодирования;	
	2. Очередь для кодированных/декодированных слайсов, ожидающих записи;

и 3 этапа фильтрации (filters):

	1. Чтение слайсов исходного файла;
	2. Кодирование/декодирование считанных слайсов;
	3. Запись обработанных слайсов в результирующий файл.

Доступ к очереди реализован в соответствии с паттерном Producer-Consumer. 
Реализован многопоточный доступ к очередям слайсов: потоки-производители добавляют слайсы в очередь, потоки-потребители извлекают.

Потоки разделены на три группы:
	1. Чтение - 1 поток для последовательного считывания файла;
        2. Кодирование/декодирование - несколько потоков: 
        	- ожидают появления слайсов в очереди на кодирование/декодирование;
        	- извлекают очереди на кодирование/декодирование;
        	- кодируют/декодируют извлеченные слайсы;
        	- добавляют кодированные/декодированные слайсы в очередь на запись.
        3. Запись - 1 поток для последовательной записи в файл.

Максимальное число создаваемых утилитой потоков было выбрано не более чем: количество_логических_ядер * 2.
То есть по 2 потока на логическое ядро - по прочитанным отзывам наилуяший вариант.
Таким образом, на кодирование/декодирование должно быть выделено: количество_логических_ядер * 2 - 2(чтение+запись).

Также, при выполнении операции чтения слайса из исходного файла, утилита осуществляет проверку на условие:
объем_свободной_оперативной_памяти > подушка_безопасности -  (резерв оперативки на другие нужды, пусть будет 500Мб).
Проверка данного условия необходима для обработки больших файлов, размер которых больше объема оперативной памяти.

Максимальное число добавляемых в каждую очередь слайсов также ограничено 
во избежание ситуаций накопления слишком большого количества слайсов 
и, соответственно, для равномерной загрузки оперативной памяти.
